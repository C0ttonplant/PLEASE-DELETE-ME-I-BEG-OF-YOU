Unit Tests

What is a unit test?

Unit tests are functions that test the input and output of a function. The most simple tests simply require giving the function being tested its inputs, and compare the output with the expected value.

This style of testing encourages functions to be pure, not using or altering the state outside of its scope. This style of programming is often referred to as MIT style. (and is personally my ideal method of programming)

Example 

                    
def mult(a: int, b: int) -> int:
    return a * b

def test(result, expected):
    if result == expected:
        print("test passed")
    else:
        print("test failed")

test(mult(1, 1), 1)  # test passed
test(mult(1, 2), 2)  # test passed
test(mult(2, 2), 4)  # test passed
test(mult(3, 2), 6)  # test passed

Problems

In an ideal world, all functions would be like this. even in the case of complex inputs and outputs, you would have the guarantee that the same inputs always give the same output. But unfortunately this isnt the case, functions can rely on global variables, and even set those variables instead of returning all its effects.

Another issue that can be run into, are functions that require a difficult to reproduce context in a testing environment, for example a function that needs a window context, or a function that needs an application context.

Example 

                    
globalvariable = True
is_equal = False

def impure(a: int, b: int) -> bool:
    global globalvariable, is_equal
    globalvariable = (a == b) and not globalvariable
    is_equal = a == b
    return globalvariable

def test(result, expected):
    if result == expected:
        print("test passed")
    else:
        print("test failed")

test(impure(1, 1), False)  # test passed
test(impure(1, 1), False)  # test failed
test(impure(2, 3), False)  # test passed
test(impure(2, 3), False)  # test failed

Creating tests

In accordance with these issues, there are good methods to use when creating tests. First, consider the boundaries of the function, whats the largest possible number, what about the most negative number. For functions that take strings, consider testing with extremely large strings, empty strings, and even null pointers. Users can and will always give inputs outside your wildest imagination.

Second, review the function, and see how all the inputs get manipulated, If its necessary for the function to set external variables, then simply add tests for how those variables are set. If a function relies on data from a database, or on external variables, creating a function to seed and reset the data will allow for easy testing. this function would also be a good place to create the necessary context, for example a dummy window context.

Example 

                    
globalvariable = True
is_equal = False

def impure(a: int, b: int) -> bool:
    global globalvariable, is_equal
    globalvariable = (a == b) and not globalvariable
    is_equal = a == b
    return globalvariable

def reset():
    global globalvariable, is_equal
    globalvariable = True
    is_equal = False

def test(result, expected):
    if result == expected:
        print("test passed")
    else:
        print("test failed")

reset()
impure(1, 1)
test(is_equal, True)   # test passed

reset()
impure(1, 1)
test(is_equal, True)   # test passed

reset()
impure(3, 2)
test(is_equal, False)  # test passed

reset()
impure(3, 2)
test(is_equal, False)  # test passed

Automated testing

Sometimes its ideal to write automatic testing, where the code can test thousands of combinations incredibly quickly, using loops and random data. Or a new form of automatic testing, using AI to generate tests, and verify functions for you.

Using loops is ideal for functions that have lots of variation in their inputs, where a large number of values can be quickly tested. On the other hand, AI can be useful for writing tedious unit tests that cant be automated. For example, functions that take objects as input.

Example 

                    
import random

def mult(a: int, b: int) -> int:
    return a * b

def test(result, expected):
    if result == expected:
        print("test passed")
    else:
        print("test failed")

for _ in range(1000):
    a = random.randint(-1000, 1000)
    b = random.randint(-1000, 1000)
    print(a, b)  # print inputs so the failing case can be identified
    test(mult(a, b), a * b)

AI

Using AI to write unit tests is a new method for good unit testing. Using AI, its easy to write many unit tests, without the tedious friction of writing the same code over and over again. AI can also be good at making unexpected input data, just like how a user would. This allows for even more comprehensive testing. AI is also good for creating seeded data for necessary databases and variables.

AI does have issues though, AI is good at generating the tests, but its not always good at generating the expected result. For example, a function that counts the number of times a char (r) appears in a string (strawberry) many AIs will give the incorrect result as well.

Example 

                    
def count(c: str, s: str) -> int:
    pass  # imagine logic here

def test(result, expected):
    if result == expected:
        print("test passed")
    else:
        print("test failed")

# GENERATED TESTS
test(count('r', "strawberry"), 4)
test(count('p', "Google"), 2)

Limitations

Unit testing is good for individual functions, using unit tests can ensure your functions are properly implemented. But unit tests cant cover how functions interact with eachother, or how a full stack application will behave. Unit tests also cant cover bugs like race conditions, memory issues, or performance bottlenecks. It doesnt matter how perfect a function is, if you use its outputs wrong.

These situations can be covered better by other testing methods, which I am excited to hear about from my classmates.

Frameworks

While its totally possible to write the testing function yourself, using a testing framework allows for much greater reporting, along with easier to produce tests. We use testing frameworks because they can be compatible with automated scripts, they can be run automatically with github actions, and it adds ease of use for other people using your code. I will go into PyTest specifically, how to set up and use.

pytest is installed as a pip module, then can be run in the command line as `pytest` this will run all files with'*_test.py test_*.py'. Inside the test file there can be multiple ways of structuring the tests, but I will go over how I find it the neatest

Example test_main.py 

                    
import pytest
# import the functions to test
from examples import mult, count

# create the test class, can be any name that starts with `Test`
class TestClass:

    # create test functions, can be any name that starts with 'test_'. parameter `self` returns `None`
    def test_mult(self) -> None:

        # assert is used as the test, this value should always resolve to a bool.
        assert mult(1, 2) == 2

        # multiple asserts can be in the same function
        assert mult(3, 4) == 12

        # for imprecise floats, `approx` can be used to compare the near value
        assert mult(0.1, 0.3) == pytest.approx(0.03)

    def test_count(self) -> None:
        assert count('r', "strawberry") == 3
        assert count('a', "apple") == 1

Solutions: Seeding & Context

For functions that rely on external state like databases or global variables, pytest fixtures provide a clean solution. A fixture is a function decorated with @pytest.fixture that sets up the necessary state before a test runs, and tears it down afterward using yield. This replaces the manual reset() approach and integrates directly with pytest's test runner.

For functions that require a window or application context, mocking is the standard solution. Python's unittest.mock library lets you create fake objects that behave like the real thing, without needing an actual window or running application. You control exactly what the fake object returns, making tests predictable and quick to run.

Example test_main.py 

                    
import pytest
from unittest.mock import MagicMock
from examples import query_database, process_window_data

# seeds the database before each test, clears it after
@pytest.fixture
def seeded_db():
    db = create_test_database()
    db.insert({"id": 1, "value": "test"})
    yield db  # test runs here
    db.clear()

# creates a fake window object with controlled values
@pytest.fixture
def mock_window():
    window = MagicMock()
    window.width = 1920
    window.height = 1080
    return window

class TestClass:

    # fixture is passed as a parameter, pytest injects it automatically
    def test_query(self, seeded_db) -> None:
        result = query_database(seeded_db, 1)
        assert result["value"] == "test"

    def test_window(self, mock_window) -> None:
        result = process_window_data(mock_window)
        assert result == (1920, 1080)
